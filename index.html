    GoodsMart Full Stack Developer · .NET | Angular 2024

    Eliminating Duplicate Charges at Scale

    A small race condition was costing real money. Here is how I closed it for the whole platform.

    Problem

    What went wrong

    Under peak traffic, the order-creation endpoint could be hit twice within a few milliseconds — a double tap on the checkout button, a flaky network retry, or two browser tabs. Application-level "has this order already been created?" checks were not fast enough to catch the second request before the first one committed. The result: duplicate orders, double charges, and support tickets.

    Approach

    How I designed the fix

    • Mapped the at-risk endpoints by reading recent incident logs — order creation, refund issuance, voucher redemption, stock decrement.
    • Designed a Redis-backed atomic lock using SET key value NX PX keyed by (user_id + endpoint + idempotency_token), with a short TTL so a crashed request cannot block the user forever.
    • Built it as one small middleware behind an [Idempotent] attribute, so any controller opts in with a single line and the lock acquire/release pattern lives in one place.
    • Added structured logging on every lock contention so we can see how often the safety net fires and tune the TTL.
    • Wrote integration tests that fire two concurrent requests and assert the second one is rejected cleanly without poisoning the first.
    Outcome

    What changed

    • Duplicate charges on the protected endpoints dropped to near zero.
    • The same pattern was reused for three other critical flows (refunds, voucher redemption, warehouse stock decrement).
    • The middleware is now the team's default approach for any "must run exactly once" endpoint.
    • Support tickets related to double-charging effectively stopped.
